# 09. Deployment & Ops Doc

# Sistem E-Learning Training Karyawan

------

## 9.1 Ringkasan Dokumen

Dokumen Deployment & Ops ini menjelaskan kebutuhan deployment, konfigurasi environment, queue, scheduler, storage, integrasi eksternal, monitoring, backup, restore, dan rollback untuk Sistem E-Learning Training Karyawan.

Dokumen ini digunakan sebagai acuan saat sistem akan dipasang ke server hosting, diperbarui, dipantau, dan dipulihkan jika terjadi masalah.

Sistem ini menggunakan pendekatan Laravel monolith dengan database MySQL dan tampilan berbasis Blade, Tailwind CSS, serta JavaScript.

Target deployment awal sistem adalah shared hosting biasa, seperti Hostinger. Namun dokumen ini juga menyiapkan catatan pengembangan lanjutan jika sistem nantinya dipindahkan ke VPS atau server yang lebih fleksibel.

------

## 9.2 Tujuan Deployment & Ops

Tujuan dokumen ini adalah:

1. Menentukan kebutuhan environment untuk menjalankan sistem.

## 9.4 Environment

| Environment | Fungsi                                      | Contoh Penggunaan   |

| Local       | Digunakan developer untuk development       | Laptop developer    |
| Staging     | Digunakan untuk uji coba sebelum production | Subdomain staging   |

### 9.4.1 Local Environment

Local environment digunakan untuk:

1. Membuat fitur baru.
2. Memperbaiki bug.
3. Menjalankan migration.
4. Menjalankan seeder awal.

6. Build frontend.

```bash
APP_ENV=local

```

Staging digunakan untuk uji coba sebelum fitur naik ke production.

Staging digunakan untuk:

1. Uji fitur baru.
2. Uji migration database.
3. Uji upload file.
4. Uji submit test.
5. Uji scoring.
6. Uji export PDF/Excel.
7. Uji tampilan responsive.

Contoh:

```bash
APP_ENV=staging
APP_DEBUG=false
```

### 9.4.3 Production Environment

Production digunakan oleh Admin dan Karyawan secara langsung.

Production wajib menggunakan:

```bash
APP_ENV=production
```

## 9.5 Server & Runtime Requirement

### 9.5.1 Target Hosting Awal

Target deployment awal:

```text
Shared hosting biasa, contoh: Hostinger
```

Catatan:

1. Shared hosting cocok untuk versi awal jika jumlah user belum terlalu besar.
2. Shared hosting memiliki keterbatasan pada queue worker, Supervisor, cron, dan akses terminal.
3. Jika sistem mulai sering digunakan dan file materi semakin besar, sistem disarankan dipindahkan ke VPS.

### 9.5.2 Runtime Requirement

| Komponen   | Versi / Kebutuhan               |
| ---------- | ------------------------------- |
| PHP        | 8.3+                            |
| Database   | MySQL 8.0+                      |
| Composer   | 2.x                             |
| Node.js    | LTS, untuk build frontend       |
| Web Server | Nginx + PHP-FPM jika tersedia   |
| Framework  | Laravel                         |
| Frontend   | Blade, Tailwind CSS, JavaScript |
| Storage    | Local private storage           |
| Queue      | Database queue                  |
| Scheduler  | Cron / Laravel scheduler        |

### 9.5.3 Web Server

Web server yang direkomendasikan:

```text
Nginx + PHP-FPM
```

Namun pada shared hosting, konfigurasi web server biasanya sudah diatur oleh provider hosting.

Jika menggunakan Hostinger, konfigurasi yang perlu diperhatikan:

1. Document root diarahkan ke folder `public`.
2. PHP version menggunakan PHP 8.3 atau lebih baru.
3. File `.env` tidak boleh berada di public path.
4. Folder `storage` dan `bootstrap/cache` harus writable.
5. HTTPS/SSL wajib aktif.

## 9.6 Environment Variable `.env`

File `.env` digunakan untuk menyimpan konfigurasi environment.

### 9.6.1 App Configuration

```env
APP_NAME="E-Learning Training Karyawan"
APP_ENV=production
APP_KEY=base64:GENERATED_APP_KEY
APP_DEBUG=false
APP_URL=https://domain-sistem.com
APP_TIMEZONE=Asia/Jakarta
```

### 9.6.2 Database Configuration

```env
DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=nama_database
DB_USERNAME=nama_user_database
DB_PASSWORD=password_database
```

### 9.6.3 Session Configuration

```env
SESSION_DRIVER=database
SESSION_LIFETIME=120
SESSION_ENCRYPT=false
SESSION_PATH=/
SESSION_DOMAIN=null
```

Catatan:

1. Session dapat menggunakan database agar lebih mudah dikelola.
2. Jika shared hosting tidak mendukung session database dengan baik, dapat menggunakan file session.
3. Session lifetime disesuaikan dengan kebutuhan perusahaan.

### 9.6.4 Cache Configuration

```env
CACHE_STORE=file
```

Catatan:

1. Untuk shared hosting, cache file cukup untuk versi awal.
2. Redis belum wajib digunakan pada versi awal.

### 9.6.5 Queue Configuration

```env
QUEUE_CONNECTION=database
```

Queue database digunakan agar sistem tetap sederhana dan tidak membutuhkan Redis.

### 9.6.6 Mail Configuration

```env
MAIL_MAILER=smtp
MAIL_HOST=smtp.example.com
MAIL_PORT=587
MAIL_USERNAME=mail@example.com
MAIL_PASSWORD=password_email
MAIL_ENCRYPTION=tls
MAIL_FROM_ADDRESS=mail@example.com
MAIL_FROM_NAME="E-Learning Training Karyawan"
```

Catatan:

1. Email notification belum aktif pada versi awal.
2. SMTP disiapkan untuk kebutuhan future integration.
3. Email dapat digunakan nanti untuk reset password, notifikasi training, atau laporan.

### 9.6.7 Storage Configuration

```env
FILESYSTEM_DISK=local
```

Storage utama menggunakan local private storage.

### 9.6.8 Logging Configuration

```env
LOG_CHANNEL=stack
LOG_LEVEL=error
```

Catatan:

1. Production menggunakan log level error.
2. Data sensitif seperti password, token, dan session tidak boleh masuk ke log.

## 9.7 Git Branching Strategy

Branch utama yang digunakan:

| Branch    | Fungsi                        |
| --------- | ----------------------------- |
| `develop` | Untuk development dan staging |
| `main`    | Untuk production              |

### 9.7.1 Alur Branch

1. Developer membuat fitur di local.
2. Fitur digabung ke branch `develop`.
3. Branch `develop` diuji di staging.
4. Jika sudah stabil, perubahan digabung ke branch `main`.
5. Server production melakukan deploy dari branch `main`.

### 9.7.2 Aturan Branch Production

Branch `main` hanya boleh berisi kode yang:

1. Sudah diuji di local.
2. Sudah diuji di staging.
3. Tidak memiliki error kritikal.
4. Tidak merusak login, training flow, submit test, scoring, dan export.
5. Sudah siap digunakan user production.

## 9.8 Deployment Flow

Deployment awal dilakukan menggunakan Git pull di server.

### 9.8.1 Deployment Awal

Langkah deployment awal:

1. Siapkan hosting dan domain.
2. Aktifkan SSL/HTTPS.
3. Buat database MySQL.
4. Upload atau clone project Laravel.
5. Arahkan document root ke folder `public`.
6. Copy file `.env`.
7. Isi konfigurasi `.env`.
8. Jalankan `composer install`.
9. Generate app key.
10. Jalankan migration.
11. Jalankan seeder awal jika diperlukan.
12. Build frontend.
13. Set permission folder.
14. Jalankan cache Laravel.
15. Lakukan smoke test.

Contoh command:

```bash
composer install --no-dev --optimize-autoloader
php artisan key:generate
php artisan migrate --force
php artisan storage:link
php artisan config:cache
php artisan route:cache
php artisan view:cache
```

Catatan:

Jika menggunakan private storage penuh, `storage:link` hanya digunakan untuk file yang memang aman diakses publik. File materi training tetap harus diakses melalui controller Laravel.

------

## 9.8.2 Deployment Update

Langkah update production:

1. Pastikan perubahan sudah diuji di staging.
2. Backup database production.
3. Aktifkan maintenance mode jika update besar.
4. Pull branch `main`.
5. Install/update dependency jika ada perubahan.
6. Jalankan migration jika ada.
7. Build atau upload hasil build frontend.
8. Clear dan cache ulang konfigurasi.
9. Nonaktifkan maintenance mode.
10. Lakukan smoke test.
11. Pantau log setelah deployment.

Contoh command:

```bash
php artisan down

git pull origin main
composer install --no-dev --optimize-autoloader
php artisan migrate --force

php artisan optimize:clear
php artisan config:cache
php artisan route:cache
php artisan view:cache

php artisan up
```

Jika tidak ada migration dan update kecil, maintenance mode dapat dilewati.

## 9.9 Queue Configuration

Queue digunakan untuk proses yang berpotensi berat atau tidak harus selesai langsung pada request utama.

### 9.9.1 Queue Driver

Queue driver yang digunakan:

```env
QUEUE_CONNECTION=database
```

### 9.9.2 Kebutuhan Queue

Queue dapat digunakan untuk:

1. Export laporan PDF/Excel besar.
2. Pengiriman email jika fitur email aktif di fase berikutnya.
3. Proses notifikasi jika dikembangkan.
4. Proses operasional lain yang berjalan di background.

Catatan:

Pada versi awal, export dapat tetap dilakukan langsung jika data belum besar. Queue disiapkan sebagai pondasi jika data bertambah.

### 9.9.3 Queue Table

Jika menggunakan database queue, jalankan:

```bash
php artisan queue:table
php artisan migrate
```

### 9.9.4 Queue Worker

Idealnya queue worker dijalankan menggunakan Supervisor.

Namun pada shared hosting, Supervisor biasanya tidak tersedia.

Opsi yang digunakan:

| Kondisi Hosting                 | Strategi                            |
| ------------------------------- | ----------------------------------- |
| Shared hosting tanpa Supervisor | Queue diproses via cron atau manual |
| Shared hosting dengan cron      | Jalankan queue command berkala      |
| VPS                             | Gunakan Supervisor                  |
| Managed server                  | Ikuti fasilitas provider            |

Contoh cron alternatif:

```bash
php artisan queue:work --stop-when-empty
```

Catatan:

1. Jika shared hosting tidak mendukung long-running process, gunakan `--stop-when-empty`.
2. Jika sistem pindah ke VPS, gunakan Supervisor agar queue worker tetap hidup.
3. Queue tidak boleh digunakan untuk proses kritikal jika worker belum stabil.

## 9.10 Scheduler Configuration

Scheduler digunakan untuk menjalankan proses rutin.

### 9.10.1 Fungsi Scheduler

Scheduler dapat digunakan untuk:

1. Backup database harian.
2. Cleanup temporary file.
3. Cleanup failed jobs lama.
4. Cleanup cache tertentu jika diperlukan.
5. Proses reminder jika fitur notifikasi dikembangkan.

### 9.10.2 Scheduler Runner

Pada Laravel standar, scheduler dijalankan setiap menit melalui cron:

```bash
* * * * * php /path-to-project/artisan schedule:run >> /dev/null 2>&1
```

### 9.10.3 Catatan Shared Hosting

Pada shared hosting, cron dapat dibatasi oleh provider.

Jika cron setiap menit tidak tersedia, alternatif:

1. Jalankan cron setiap 5 menit.
2. Jalankan backup melalui fitur backup hosting.
3. Jalankan command maintenance secara manual jika proses tidak kritikal.

### 9.10.4 Scheduler Scope Awal

Scheduler versi awal cukup digunakan untuk:

1. Backup database.
2. Cleanup temporary file.
3. Maintenance ringan.

Scheduler tidak digunakan untuk fitur kompleks pada versi awal.

## 9.11 Storage & File Access

Storage digunakan untuk menyimpan materi training dan file pendukung sistem.

### 9.11.1 Jenis File Materi

Sistem mendukung materi berupa:

1. PDF.
2. PPT/PPTX.
3. DOC/DOCX.
4. XLS/XLSX.
5. CSV.
6. JPG.
7. PNG.
8. JPEG.
9. WEBP.
10. MP4.
11. Link Google Drive.
12. Link YouTube private/unlisted.

### 9.11.2 Storage Strategy

Strategi storage:

```text
Private local storage
```

File materi tidak boleh diakses langsung secara publik tanpa login.

### 9.11.3 Lokasi Penyimpanan

Contoh struktur storage:

```text
storage/app/private/materials
storage/app/private/exports
storage/app/private/temp
```

### 9.11.4 Akses File Materi

Akses file dilakukan melalui route/controller Laravel.

Contoh flow:

1. Karyawan login.
2. Karyawan membuka detail training.
3. Sistem memeriksa apakah karyawan memiliki assignment training.
4. Sistem memeriksa apakah materi termasuk training tersebut.
5. Sistem mencatat akses materi.
6. Sistem menampilkan atau mengunduh file.

Contoh route:

```text
GET /training/{training}/materials/{material}/view
GET /training/{training}/materials/{material}/download
```

### 9.11.5 Validasi Upload File

Validasi upload wajib mencakup:

1. Ekstensi file.
2. MIME type jika memungkinkan.
3. Ukuran file.
4. Nama file.
5. Hak akses upload hanya Admin.
6. File tidak boleh dieksekusi sebagai script.

### 9.11.6 Batas Ukuran File

Batas awal ukuran file:

```text
50 MB per file
```

Catatan:

1. Untuk dokumen dan PPT, 50 MB umumnya cukup.
2. Untuk video besar, lebih disarankan menggunakan link YouTube private/unlisted atau Google Drive.
3. Jika sering upload video besar, sistem sebaiknya dipindah ke VPS atau cloud storage.

### 9.11.7 Delete File

Jika materi dihapus permanen:

1. Record materi di database dihapus.
2. File fisik di storage ikut dihapus.
3. Sistem mencatat aktivitas delete jika audit trail tersedia.
4. Sistem menampilkan notifikasi berhasil.


```bash

```

### 9.15.3 Waktu Maintenance

Maintenance dilakukan:

```text
Di luar jam kerja.
```

## 9.16 Rollback Plan

Rollback digunakan jika deployment gagal atau sistem mengalami error setelah update.

### 9.16.1 Jenis Rollback

Rollback dibagi menjadi:

1. Rollback code.
2. Rollback database.
3. Rollback file storage.
4. Rollback konfigurasi.

### 9.16.2 Rollback Code

Rollback code dilakukan dengan kembali ke commit atau release sebelumnya.

Langkah umum:

1. Aktifkan maintenance mode.
2. Cek commit terakhir yang stabil.
3. Checkout ke commit stabil.
4. Install dependency jika diperlukan.
5. Clear cache.
6. Test sistem.
7. Nonaktifkan maintenance mode.

Contoh:

```bash
php artisan down
git log --oneline
git checkout <commit_stabil>
composer install --no-dev --optimize-autoloader
php artisan optimize:clear
php artisan config:cache
php artisan route:cache
php artisan view:cache
php artisan up
```

### 9.16.3 Rollback Database

Rollback database dilakukan jika migration atau perubahan data menyebabkan error.

Langkah:

1. Aktifkan maintenance mode.
2. Cek migration terakhir.
3. Jalankan rollback migration jika aman.
4. Jika tidak aman, restore dari backup database.
5. Test data penting.
6. Nonaktifkan maintenance mode.

Contoh:

```bash
php artisan migrate:rollback
```

Catatan:

Rollback migration tidak selalu cukup. Jika data sudah berubah atau hilang, restore backup lebih aman.

### 9.16.4 Rollback File Storage

Rollback file storage dilakukan jika file materi hilang, rusak, atau salah upload.

Langkah:

1. Ambil backup file storage terakhir.
2. Upload ulang file ke path yang benar.
3. Pastikan permission folder benar.
4. Test akses file dari sistem.
5. Cek log error file.

### 9.16.5 Rollback `.env`

Rollback `.env` dilakukan jika konfigurasi production salah.

Langkah:

1. Gunakan backup `.env` sebelumnya.
2. Pastikan database, app URL, mail, queue, dan storage benar.
3. Jalankan:

```bash
php artisan config:clear
php artisan config:cache
```

1. Test login dan fitur utama.

## 9.17 Seeder Production

Seeder di production hanya digunakan untuk kebutuhan awal.

Seeder yang boleh dijalankan:

1. Initial admin.
2. Data konfigurasi default jika ada.
3. Data master awal jika benar-benar diperlukan.

Seeder yang tidak boleh dijalankan sembarangan:

1. Dummy karyawan.
2. Dummy training.
3. Dummy soal.
4. Dummy attempt.
5. Dummy jawaban test.
6. Dummy progress.

Catatan:

Seeder production harus aman dan tidak menghapus data existing.

## 9.18 Cache & Optimization

Command yang digunakan:

```bash

php artisan route:cache
php artisan view:cache
```

Catatan:

1. Jalankan `optimize:clear` setelah update.
2. Jalankan cache ulang setelah konfigurasi benar.
3. Pastikan route tidak menggunakan closure jika ingin memakai `route:cache`.

## 9.19 File Permission

Folder yang harus writable:

```text
storage
bootstrap/cache
```

Permission tidak boleh dibuat terlalu terbuka.

Rekomendasi:

1. Hindari permission `777` jika tidak perlu.
2. Gunakan permission sesuai standar hosting.
3. Pastikan owner file sesuai user hosting.
4. Pastikan Laravel dapat menulis log.
5. Pastikan Laravel dapat menulis cache.
6. Pastikan Laravel dapat menyimpan file upload.

Jika terjadi error permission, cek:

1. `storage/logs`
2. `storage/framework`
3. `storage/app`
4. `bootstrap/cache`

## 9.23 Risiko Operasional

| Risiko                                             | Dampak                               | Mitigasi                                                     |
| -------------------------------------------------- | ------------------------------------ | ------------------------------------------------------------ |
| Shared hosting tidak mendukung queue worker stabil | Job background tidak berjalan        | Gunakan cron `queue:work --stop-when-empty` atau proses langsung |
| Shared hosting membatasi cron                      | Scheduler tidak berjalan tepat waktu | Gunakan cron sesuai batas hosting atau backup manual         |
| File materi terlalu besar                          | Storage cepat penuh                  | Batasi upload 50 MB dan gunakan link video                   |
| Backup hanya di server yang sama                   | Data hilang jika server rusak        | Simpan salinan backup ke external storage                    |
| Migration gagal                                    | Sistem error                         | Backup database sebelum migrate                              |
| Error production tidak terlihat cepat              | Perbaikan lambat                     | Cek log harian dan buat prosedur pelaporan error             |
| File permission salah                              | Upload/log/cache gagal               | Pastikan `storage` dan `bootstrap/cache` writable            |
| APP_DEBUG aktif di production                      | Error teknis bocor ke user           | Wajib `APP_DEBUG=false`                                      |

## 9.24 Rekomendasi Upgrade Infrastruktur

Jika sistem mulai besar, disarankan upgrade dari shared hosting ke VPS.

### 9.24.1 Indikator Perlu Upgrade ke VPS

Upgrade ke VPS jika:

1. Jumlah user aktif meningkat.
2. File materi semakin besar.
3. Export laporan mulai lambat.
4. Queue worker dibutuhkan secara stabil.
5. Scheduler harus berjalan rutin.
6. Backup perlu otomatis lebih rapi.
7. Storage hosting mulai penuh.
8. Sistem mulai sering digunakan untuk training penting.

### 9.24.2 Infrastruktur VPS yang Disarankan

Jika pindah ke VPS, gunakan:

1. Ubuntu Server.
2. Nginx.
3. PHP-FPM.
4. MySQL 8.0.
5. Supervisor untuk queue worker.
6. Cron untuk Laravel scheduler.
7. Backup otomatis ke external storage.
8. Monitoring error tambahan jika diperlukan.
