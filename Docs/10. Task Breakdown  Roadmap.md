# 10. Task Breakdown / Roadmap

# Sistem E-Learning Training Karyawan

------

## 10.1 Ringkasan Dokumen

Dokumen **Task Breakdown / Roadmap** ini menjelaskan pembagian pekerjaan pengembangan Sistem E-Learning Training Karyawan ke dalam bentuk milestone, epic, story, task teknis, dependency, prioritas, dan status progress.

Dokumen ini digunakan sebagai acuan untuk:

1. Membagi pekerjaan development menjadi bagian yang lebih kecil dan mudah dikerjakan.
2. Menentukan urutan pengerjaan fitur berdasarkan dependency.
3. Menentukan prioritas fitur yang wajib selesai lebih dulu.
4. Menjadi checklist progress development.
5. Menjadi acuan developer, QA, dan project owner untuk memantau status pengerjaan.
6. Memastikan fitur yang dibangun sesuai dengan PRD, FRD, User Flow, ERD, UI Spec, Acceptance Criteria, Test Plan, NFR, dan Deployment & Ops.

Roadmap ini dibuat untuk pengembangan sistem berbasis **Laravel monolith**, menggunakan **MySQL**, **Blade**, **Tailwind CSS**, dan **JavaScript**.

------

## 10.2 Tujuan Task Breakdown / Roadmap

Tujuan dokumen ini adalah:

1. Menyusun daftar pekerjaan pengembangan sistem secara terstruktur.
2. Membagi pekerjaan ke dalam milestone besar: MVP, Beta, dan Production.
3. Menentukan epic utama yang harus dikerjakan.
4. Menentukan story berdasarkan kebutuhan Admin dan Karyawan.
5. Menentukan task teknis yang perlu dikerjakan oleh developer.
6. Menentukan dependency antar fitur.
7. Menentukan prioritas task menggunakan P1, P2, dan P3.
8. Menyediakan format checklist tracking progress.
9. Membantu proses testing per fitur dan regression test.
10. Menjadi dasar untuk proses deployment ke staging dan production.

------

## 10.3 Prinsip Penyusunan Roadmap

Prinsip penyusunan roadmap:

1. Roadmap disusun berdasarkan urutan pengerjaan teknis.
2. Task tetap dikelompokkan berdasarkan modul agar mudah dipahami.
3. Fitur dengan dependency besar dikerjakan setelah fondasi sistem siap.
4. Authentication dan role access dikerjakan di awal.
5. Database, migration, seeder, index, constraint, dan sample data masuk ke roadmap.
6. UI component reusable dibuat di awal agar tampilan konsisten.
7. Testing dilakukan per fitur setelah fitur selesai.
8. Regression test dilakukan di akhir sebelum production.
9. Security dan NFR dibuat sebagai epic khusus serta checklist per fitur.
10. Deployment menjadi milestone terakhir sebelum sistem digunakan oleh user asli.

------

## 10.4 Format Status, Prioritas, dan Dependency

### 10.4.1 Status Progress

| Status         | Keterangan                                                   |
| -------------- | ------------------------------------------------------------ |
| Not Started    | Task belum dikerjakan                                        |
| In Progress    | Task sedang dikerjakan                                       |
| Ready for Test | Task selesai dibuat dan siap diuji                           |
| Testing        | Task sedang diuji                                            |
| Revision       | Task perlu diperbaiki karena ada bug atau belum sesuai requirement |
| Done           | Task selesai, sudah diuji, dan dinyatakan sesuai             |

------

### 10.4.2 Prioritas

| Prioritas | Keterangan                           | Contoh                                                     |
| --------- | ------------------------------------ | ---------------------------------------------------------- |
| P1        | Fitur kritikal yang wajib berjalan   | Login, role access, submit test, scoring, status kelulusan |
| P2        | Fitur penting pendukung proses utama | Upload materi, search, filter, export laporan              |
| P3        | Fitur pendukung UI dan kenyamanan    | Empty state, loading state, minor UI polish                |

------

### 10.4.3 Dependency

Dependency adalah task atau fitur yang harus selesai terlebih dahulu sebelum task lain dapat dikerjakan.

Contoh:

| Task               | Dependency                                                   |
| ------------------ | ------------------------------------------------------------ |
| CRUD Karyawan      | Tabel users, employees, divisions, positions                 |
| CRUD Materi        | CRUD Training                                                |
| CRUD Soal          | CRUD Training                                                |
| Penugasan Training | Data Training dan Karyawan                                   |
| Submit Post-Test   | Training assigned, materi sudah dibuka, soal post-test tersedia |
| Export Laporan     | Progress dan scoring sudah tersedia                          |

# 10.12 Definition of Done

Sebuah task dinyatakan **Done** jika memenuhi kondisi berikut:

1. Fitur sudah selesai dikembangkan.
2. Fitur sudah sesuai dengan requirement.
3. Validasi form berjalan.
4. Role access sesuai.
5. UI sesuai layout dan komponen reusable.
6. Empty, error, dan success state tersedia jika dibutuhkan.
7. Data tersimpan dengan benar ke database.
8. Tidak ada error utama pada flow normal.
9. Sudah diuji berdasarkan Acceptance Criteria.
10. Sudah diuji berdasarkan QA Checklist.
11. Bug P1 sudah diperbaiki.
12. Fitur sudah masuk branch yang sesuai.
13. Jika berhubungan dengan production, sudah lolos deployment checklist.

# 10.13 Risiko dan Catatan Roadmap

| Risiko                          | Dampak                                | Mitigasi                                                    |
| ------------------------------- | ------------------------------------- | ----------------------------------------------------------- |
| Scope fitur terlalu besar       | Development menjadi lama              | Fokus MVP terlebih dahulu                                   |
| Upload file besar               | Storage hosting cepat penuh           | Batasi ukuran file dan monitoring storage                   |
| Shared hosting terbatas         | Queue/scheduler tidak optimal         | Gunakan database queue sederhana atau pindah VPS jika perlu |
| Validasi role terlewat          | Karyawan bisa akses halaman Admin     | Wajib test role access semua route                          |
| Scoring salah                   | Status lulus/tidak lulus tidak akurat | Buat test case scoring detail                               |
| Essay belum dinilai             | Status training menggantung           | Tampilkan daftar jawaban essay yang menunggu penilaian      |
| Export berat                    | Server lambat                         | Export mengikuti filter dan batasi data                     |
| Migration production gagal      | Sistem error saat deploy              | Backup database sebelum migration                           |
| File materi bisa diakses publik | Risiko keamanan data                  | Simpan file di private storage dan akses via controller     |
| UI tidak konsisten              | Pengalaman pengguna buruk             | Gunakan reusable component sejak awal                       |
