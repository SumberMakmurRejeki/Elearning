# 08. Test Plan / QA Checklist

# Sistem E-Learning Training Karyawan

------

## 8.1 Ringkasan Dokumen

Dokumen Test Plan / QA Checklist ini digunakan sebagai acuan pengujian Sistem E-Learning Training Karyawan.

Dokumen ini berisi skenario pengujian yang disusun secara to the point berdasarkan fitur utama sistem, alur pengguna, acceptance criteria, dan rancangan UI.

Fokus utama dokumen ini adalah:

1. Menentukan skenario test per modul.
2. Menentukan happy path untuk alur normal sistem.
3. Menentukan edge case untuk kondisi tidak normal atau data tidak valid.
4. Menentukan regression checklist untuk memastikan fitur penting tetap berjalan setelah ada perubahan.
5. Menentukan prioritas pengujian agar QA dapat fokus pada fitur paling kritikal.
6. Menjadi acuan developer dan tester sebelum fitur dinyatakan selesai.

------

## 8.2 Tujuan Testing

Tujuan testing adalah:

1. Memastikan fitur berjalan sesuai kebutuhan sistem.
2. Memastikan alur Admin dan Karyawan berjalan dari awal sampai akhir.
3. Memastikan validasi form berjalan sesuai aturan.
4. Memastikan akses role Admin dan Karyawan tidak tertukar.
5. Memastikan proses training, materi, test, scoring, dan laporan berjalan benar.
6. Memastikan UI state seperti empty, loading, error, success, dan confirmation berjalan konsisten.
7. Memastikan fitur penting tetap aman saat ada update atau perbaikan sistem.

------

## 8.3 Scope Testing

### 8.3.1 Scope In

Pengujian mencakup:

1. Authentication dan role access.
2. Dashboard Admin dan Karyawan.
3. Master Data Karyawan, Divisi, dan Jabatan.
4. Manajemen Training.
5. Materi Training.
6. Soal Pre-Test dan Post-Test.
7. Penugasan Training.
8. Alur Karyawan mengikuti training.
9. Pre-Test dan Post-Test.
10. Penilaian pilihan ganda otomatis.
11. Penilaian essay manual.
12. Perhitungan nilai akhir.
13. Status lulus, tidak lulus, dan menunggu penilaian.
14. Pengulangan post-test.
15. Monitoring progress training.
16. Laporan dan export PDF/Excel.
17. Profil dan ubah password.
18. UI state dan responsive layout.

### 8.3.2 Scope Out

Pengujian tidak mencakup:

1. Multi admin.
2. Role permission detail.
3. Sertifikat otomatis.
4. Notifikasi email.
5. Import karyawan dari Excel.
6. Random soal.
7. Timer test.
8. Anti-cheat.
9. Forum diskusi.
10. Live class.
11. Tracking video berdasarkan persentase tontonan.
12. Integrasi Google Drive API.
13. Payment.
14. Mobile app native Android/iOS.

------

## 8.4 Jenis Testing

| Jenis Test  | Keterangan                                                   | Contoh                                                       |
| ----------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| Happy Path  | Menguji alur normal sistem                                   | Admin berhasil membuat training dan karyawan menyelesaikan post-test |
| Edge Case   | Menguji kondisi tidak normal                                 | Username kosong, file gagal upload, session expired          |
| Regression  | Menguji ulang fitur penting setelah ada perubahan            | Login, training flow, scoring, laporan                       |
| UI State    | Menguji empty, loading, error, success, dan confirmation state | Data kosong menampilkan empty state                          |
| Role Access | Menguji pembatasan akses berdasarkan role                    | Karyawan tidak boleh akses dashboard admin                   |

------

## 8.5 Prioritas Testing

| Prioritas | Keterangan                                 | Contoh                                        |
| --------- | ------------------------------------------ | --------------------------------------------- |
| P1        | Fitur kritikal yang wajib berjalan         | Login, submit test, scoring, status kelulusan |
| P2        | Fitur penting pendukung proses utama       | Search, filter, export, upload materi         |
| P3        | Fitur pendukung UI dan kenyamanan pengguna | Empty state, loading state, tampilan minor    |

------

## 8.6 Status Testing

| Status     | Keterangan                                                 |
| ---------- | ---------------------------------------------------------- |
| Not Tested | Test belum dilakukan                                       |
| Pass       | Test berhasil sesuai expected result                       |
| Fail       | Test gagal atau hasil tidak sesuai                         |
| Blocked    | Test tidak bisa dilakukan karena ada dependensi/error lain |
| Retest     | Test perlu diuji ulang setelah perbaikan                   |

## 8.8 Format QA Checklist

Format tabel QA Checklist:

| Kolom           | Keterangan                                               |
| --------------- | -------------------------------------------------------- |

| Referensi AC    | Referensi acceptance criteria jika ada                   |

| Skenario Test   | Nama skenario pengujian                                  |

# 8.13 Exit Criteria

Fitur dapat dinyatakan selesai jika memenuhi kriteria berikut:

1. Semua test case prioritas P1 berstatus Pass.
2. Minimal test case prioritas P2 untuk fitur yang diuji berstatus Pass.
3. Tidak ada bug critical pada login, role access, test, scoring, dan status kelulusan.
4. Tidak ada error utama pada proses submit pre-test dan post-test.
5. Data penting tersimpan dengan benar di database.
6. Nilai pilihan ganda otomatis sesuai jawaban benar dan bobot nilai.
7. Nilai essay dapat diinput oleh Admin dan dihitung ke nilai akhir.
8. Status lulus/tidak lulus sesuai passing grade.
9. Export PDF dan Excel berjalan untuk laporan utama.
10. UI utama dapat digunakan di desktop, tablet, dan mobile.
11. Empty state, loading state, error state, success notification, dan confirmation modal berjalan sesuai kebutuhan.
12. Regression test wajib sudah dilakukan setelah perubahan fitur penting.

# 8.16 Catatan Implementasi QA

1. Testing dilakukan minimal pada browser modern seperti Chrome, Firefox, dan Edge.
2. Testing responsive dilakukan pada desktop, tablet, dan mobile.
3. Data test sebaiknya dipisahkan dari data production.
4. Aksi delete permanen harus diuji dengan hati-hati.
5. Semua aksi berisiko harus menggunakan custom modal.
6. Fitur test, scoring, dan status kelulusan menjadi prioritas utama.
7. Setiap bug P1 wajib diperbaiki sebelum fitur dinyatakan selesai.
8. Regression test wajib dilakukan setelah perubahan pada fitur training, test, scoring, dan laporan.
