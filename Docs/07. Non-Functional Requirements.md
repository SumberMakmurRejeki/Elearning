# 07. Non-Functional Requirements

# Sistem E-Learning Training Karyawan

------

## 7.1 Ringkasan Dokumen

Dokumen Non-Functional Requirements atau NFR ini menjelaskan kebutuhan non-fungsional pada Sistem E-Learning Training Karyawan.

Dokumen ini digunakan sebagai acuan untuk memastikan sistem tidak hanya berjalan secara fungsi, tetapi juga aman, stabil, cukup cepat, mudah dipelihara, dapat diaudit, memiliki backup, serta nyaman digunakan pada berbagai perangkat.

Non-Functional Requirements mencakup:

1. Security.
2. Performance.
3. Logging.
4. Backup.
5. Roles & permissions.
6. Audit trail.
7. File upload dan storage.
8. Error handling.
9. Browser dan device support.
10. Accessibility.
11. Availability dan maintenance.
12. Data retention.

------

## 7.2 Tujuan Non-Functional Requirements

Tujuan dokumen ini adalah:

1. Menentukan standar keamanan sistem.
2. Menentukan batas performa sistem.
3. Menentukan aturan logging dan error handling.
4. Menentukan kebutuhan backup dan recovery.
5. Menentukan aturan akses berdasarkan role.
6. Menentukan aktivitas penting yang perlu dicatat dalam audit trail.
7. Menentukan standar file upload dan penyimpanan materi.
8. Menentukan dukungan browser dan device.
9. Menentukan standar kualitas UI dan accessibility.
10. Menjadi acuan developer dan QA dalam proses implementasi serta pengujian sistem.

------

## 7.3 Scope NFR

### 7.3.1 Scope In

Non-Functional Requirements mencakup:

1. Autentikasi login Admin dan Karyawan.
2. Pengamanan password.
3. Session login.
4. Pembatasan akses berdasarkan role.
5. Validasi akses route backend.
6. Audit trail untuk aksi penting.
7. Logging error dan aktivitas penting.
8. Target performa halaman dan proses utama.
9. Pencegahan double submit pada test.
10. Validasi file upload materi.
11. Penyimpanan file materi dengan akses terkontrol.
12. Backup database dan file upload.
13. Recovery dari backup terakhir.
14. Browser support.
15. Device support.
16. Custom error page.
17. UI consistency.
18. Basic accessibility.
19. Availability pada jam kerja.
20. Maintenance di luar jam kerja.
21. Data retention selama sistem digunakan.

### 7.3.2 Scope Out

Non-Functional Requirements versi awal tidak mencakup:

1. Multi admin.
2. Permission detail berbasis tabel `roles` dan `permissions`.
3. Captcha login.
4. Pembatasan gagal login sementara.
5. Two-factor authentication.
6. Single sign-on.
7. Monitoring eksternal seperti Sentry, Bugsnag, Grafana, atau Loki.
8. High availability dengan SLA 99.9%.
9. Background job untuk export.
10. Cloud storage wajib.
11. Auto delete data berdasarkan masa retensi tertentu.
12. Mobile app native Android/iOS.

# 7.4 Security Requirements

## 7.4.1 Prinsip Security

Security sistem menggunakan pendekatan sederhana tetapi tetap aman untuk sistem internal perusahaan.

Prinsip security:

1. Setiap user wajib login sebelum mengakses sistem.
2. Password tidak boleh disimpan dalam bentuk plain text.
3. Akses halaman wajib dibatasi berdasarkan role.
4. File materi tidak boleh dapat diakses publik tanpa login.
5. Input user wajib divalidasi.
6. Sistem wajib melindungi form dari request tidak sah.
7. Error teknis tidak boleh ditampilkan langsung ke user.
8. Data sensitif tidak boleh masuk ke log.

# 7.5 Roles & Permissions Requirements

## 7.5.1 Prinsip Role Access

Sistem menggunakan dua role utama:

1. Admin.
2. Karyawan.

Role dibuat sederhana dan tidak menggunakan permission detail pada versi awal.

Admin memiliki akses untuk mengelola data master, training, materi, soal, penugasan, penilaian, monitoring, laporan, export, profil, dan password.

Karyawan memiliki akses untuk melihat training yang ditugaskan, mengakses materi, mengerjakan test, melihat status, melihat nilai jika diizinkan, melihat riwayat, profil, dan ubah password.

# 7.6 Audit Trail Requirements

## 7.6.1 Prinsip Audit Trail

Audit trail digunakan untuk mencatat aktivitas penting yang berdampak pada data, nilai, progress, materi, training, dan laporan.

Audit trail tidak mencatat semua klik user. Sistem hanya mencatat aksi penting agar data tetap ringan dan mudah dibaca.

## 7.6.2 Aktivitas yang Dicatat

Aktivitas yang perlu dicatat:

1. Login user.
2. Logout user.
3. Tambah data penting.
4. Edit data penting.
5. Delete permanen data.
6. Nonaktifkan data.
7. Aktifkan data.
8. Publish training.
9. Assign training.
10. Upload materi.
11. Delete materi.
12. Submit pre-test.
13. Submit post-test.
14. Penilaian essay oleh Admin.
15. Perubahan nilai essay.
16. Export laporan.
17. Reset password karyawan.

## 7.6.3 Data Audit Trail

Data minimal audit trail:

1. User ID.
2. Nama user.
3. Role user.
4. Aktivitas.
5. Modul.
6. Data yang terdampak.
7. Waktu aktivitas.
8. IP address jika tersedia.
9. User agent jika tersedia.
10. Status aktivitas berhasil atau gagal.

# 7.7 Logging & Error Handling Requirements

## 7.7.1 Prinsip Logging

Logging digunakan untuk membantu developer menganalisis error dan aktivitas penting sistem.

Log tidak boleh menampilkan data sensitif seperti password, token, session, atau isi jawaban user secara mentah.

## 7.7.2 Aktivitas Error yang Perlu Dicatat

Aktivitas error yang perlu masuk log:

1. Gagal login karena error sistem.
2. Gagal simpan data.
3. Gagal upload materi.
4. Gagal download materi.
5. Gagal submit test.
6. Gagal scoring otomatis.
7. Gagal penilaian essay.
8. Gagal export PDF.
9. Gagal export Excel.
10. Error query laporan.
11. Error akses file.
12. Error server.

## 7.7.3 Error Handling

Error handling wajib menggunakan pesan yang mudah dipahami user.

Contoh:

| Kondisi              | Pesan User-Friendly                                     |
| -------------------- | ------------------------------------------------------- |
| Gagal menyimpan data | Data gagal disimpan. Silakan coba lagi.                 |
| Gagal upload file    | File gagal diupload. Periksa format dan ukuran file.    |
| Session expired      | Sesi Anda telah berakhir. Silakan login kembali.        |
| Akses ditolak        | Anda tidak memiliki akses ke halaman ini.               |
| Data tidak ditemukan | Data tidak ditemukan atau sudah tidak tersedia.         |
| Error server         | Terjadi kesalahan pada sistem. Silakan coba lagi nanti. |

## 7.7.4 Custom Error Page

Sistem wajib menyediakan custom error page minimal untuk:

1. 403 Forbidden.
2. 404 Not Found.
3. 500 Server Error.

# 7.9 File Upload & Storage Requirements

## 7.9.1 Prinsip File Upload

File materi training harus divalidasi sebelum disimpan.

Sistem hanya menerima file sesuai format yang dibutuhkan dalam scope sistem.

Materi juga dapat berupa link, seperti Google Drive atau YouTube private/unlisted.

## 7.9.2 Format File yang Diizinkan

Format file materi yang diizinkan:

1. PDF.
2. PPT.
3. PPTX.
4. DOC.
5. DOCX.
6. XLS.
7. XLSX.
8. CSV.
9. JPG.
10. PNG.
11. JPEG.
12. WEBP.
13. MP4.
14. Link Google Drive.
15. Link YouTube private/unlisted.
16. Link materi lainnya jika sesuai kebutuhan training.

Format file yang tidak diizinkan:

1. EXE.
2. BAT.
3. CMD.
4. SH.
5. APK.
6. ZIP jika tidak diperlukan.
7. RAR jika tidak diperlukan.
8. File script berbahaya.
9. File dengan ekstensi tidak dikenal.

## 7.9.3 Batas Ukuran File

Batas ukuran file versi awal:

| Jenis Materi | Format                                    | Maksimal Ukuran               |
| ------------ | ----------------------------------------- | ----------------------------- |
| Dokumen      | PDF, DOC, DOCX, PPT, PPTX, XLS, XLSX, CSV | 20 MB per file                |
| Gambar       | JPG, PNG, JPEG, WEBP                      | 10 MB per file                |
| Video        | MP4                                       | 200 MB per file               |
| Link         | Google Drive, YouTube, link materi        | Tidak menggunakan upload file |

## 7.9.4 Storage

File upload disimpan menggunakan Laravel storage dengan akses terkontrol.

File tidak boleh langsung dibuka secara publik tanpa proses validasi login dan validasi akses training.
